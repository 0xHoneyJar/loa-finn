## Billing Invariant Failure Modes

<!-- provenance: CODE-FACTUAL -->
<!-- evidence: file=src/hounfour/billing-invariants.ts:28-45, file=src/hounfour/billing-finalize-client.ts:107-130, file=src/billing/circuit-breaker.ts:46-180 -->

### INV-1: COMPLETENESS

| Field | Description |
|-------|-------------|
| **Violation** | `finalize()` returns a status not in `{finalized, idempotent, dlq}` |
| **Detection** | `assertCompleteness()` throws `"INV-1 violated"` (`billing-invariants.ts:28-38`) |
| **Recovery** | Caught by outer try/catch in `finalize()` → entry routed to DLQ with `internal_error` reason (`billing-finalize-client.ts:125-128`) |
| **Blast radius** | Single billing entry. No cascade — other entries unaffected. |

### INV-2/2D: PERSISTENCE (Durable / Degraded)

| Field | Description |
|-------|-------------|
| **Violation** | DLQ entry lost (Redis crash without AOF, or in-memory store process exit) |
| **Detection** | AOF verification at boot (`src/index.ts:205-206`); degraded mode logged at boot (`src/index.ts:208`) |
| **Recovery** | Durable: Redis AOF replays entries. Degraded: entries lost on process exit — ERROR logged for manual reconciliation |
| **Blast radius** | Degraded mode: all in-flight DLQ entries at process exit. Durable mode: bounded by AOF fsync interval. |

### INV-3: IDEMPOTENCY

| Field | Description |
|-------|-------------|
| **Violation** | Duplicate finalize charges tenant twice |
| **Detection** | Arrakis returns HTTP 409 on duplicate `reservation_id`; client maps to `idempotent` (`billing-finalize-client.ts:297-299`) |
| **Recovery** | Automatic — 409 → `{ ok: true, status: "idempotent" }`. No DLQ entry created. |
| **Blast radius** | None — idempotency is the happy path for duplicate detection. |

### INV-4: COST IMMUTABILITY

| Field | Description |
|-------|-------------|
| **Violation** | `actual_cost_micro` modified after initial computation |
| **Detection** | Structural — `state-machine.ts:233-263` sets cost once during COMMITTED transition. No mutable reference exposed. |
| **Recovery** | Voided entries never reuse cost values. No runtime recovery needed — violation requires code change. |
| **Blast radius** | Single billing entry's cost record. |

### INV-5: BOUNDED RETRY

| Field | Description |
|-------|-------------|
| **Violation** | DLQ entry replayed more than `maxRetries` (default 5) times |
| **Detection** | `assertBoundedRetry()` at `billing-invariants.ts:41-44`; `replayDeadLetters()` checks `attempt_count >= maxRetries` before replay (`billing-finalize-client.ts:169-174`) |
| **Recovery** | Terminal drop: entry removed from DLQ with ERROR log including `reservation_id`, `tenant_id`, `actual_cost_micro`, `created_at`, `attempts` |
| **Blast radius** | Single DLQ entry. Terminal drop means cost is never finalized — requires manual reconciliation. |

## WAL Invariant Failure Modes

<!-- provenance: CODE-FACTUAL -->
<!-- evidence: file=src/persistence/wal.ts:43-115, file=src/billing/state-machine.ts:46-66 -->

### Monotonic Sequence Numbers

| Field | Description |
|-------|-------------|
| **Violation** | Sequence numbers non-monotonic (duplicate or out-of-order ULID) |
| **Detection** | `monotonicFactory()` at `wal.ts:18` guarantees monotonicity within a process. Cross-process: timestamp + process ID ordering on replay. |
| **Recovery** | No runtime recovery — `monotonicFactory()` is structurally monotonic. Multi-process gaps detected as verifiable discontinuity. |
| **Blast radius** | Replay ordering may be ambiguous if system clock jumps backward. |

### CRC32 Checksum Validation

| Field | Description |
|-------|-------------|
| **Violation** | WAL entry payload corrupted (checksum mismatch on replay) |
| **Detection** | SHA-256 checksum computed at append (`state-machine.ts:46-51`), verified on replay |
| **Recovery** | Corrupted entry skipped during replay with ERROR log. Recovery cascade falls back to R2/git sources. |
| **Blast radius** | Single WAL entry. Uncorrupted entries before/after are unaffected. |

### Disk Pressure Hysteresis

| Field | Description |
|-------|-------------|
| **Violation** | Disk full — WAL cannot append |
| **Detection** | `DiskPressureError` thrown when free space < 100MB (`wal.ts:84-87`) |
| **Recovery** | WAL enters read-only mode. Resumes at 150MB free (hysteresis prevents thrashing at boundary, `wal.ts:76-82`). Billing entries route to DLQ during pressure. |
| **Blast radius** | All WAL writes blocked. Billing falls back to DLQ. Sessions cannot persist new messages. |

### WAL Authoritativeness

| Field | Description |
|-------|-------------|
| **Violation** | Redis state diverges from WAL (Redis treated as authoritative) |
| **Detection** | Structural — all state transitions write WAL first, then update Redis (`state-machine.ts:1-6`). Redis is derived. |
| **Recovery** | On restart, WAL replay rebuilds Redis state. No runtime recovery needed — violation requires code change that bypasses WAL. |
| **Blast radius** | If violated: full billing state corruption. Prevented by architectural constraint. |

## Auth Contract Failure Modes

<!-- provenance: CODE-FACTUAL -->
<!-- evidence: file=src/hounfour/jwt-auth.ts:29-51, file=src/hounfour/jti-replay.ts:30-37, file=src/index.ts:486-491 -->

### JTI Replay Protection

| Field | Description |
|-------|-------------|
| **Violation** | Same JWT `jti` used twice within replay window |
| **Detection** | `JtiReplayGuard.checkAndStore()` returns `true` if jti already seen (`jti-replay.ts:44`). TTL derived from JWT exp + clock skew (`jti-replay.ts:30-37`). |
| **Recovery** | Request rejected with 401. In-memory guard: periodic sweep every 60s + max 100K entries with oldest-first eviction (`jti-replay.ts:13-14`). |
| **Blast radius** | Single request rejected. No state corruption. |

### ES256 Production Enforcement

| Field | Description |
|-------|-------------|
| **Violation** | HS256 JWT presented in production |
| **Detection** | `isProduction && explicitAlg === "HS256"` check at `src/index.ts:486-491` — throws before server starts |
| **Recovery** | Process exits with error. Operator must configure ES256 JWKS. No graceful fallback — fail-fast is intentional. |
| **Blast radius** | Entire service fails to start. Prevents insecure algorithm downgrade. |

## Credit Conservation Failure Modes

<!-- provenance: CODE-FACTUAL -->
<!-- evidence: file=src/credits/credit-persistence.ts:44-46, file=src/credits/consumption.ts:72-78, file=src/credits/nonce-cleanup.ts -->

### Sum Invariant

| Field | Description |
|-------|-------------|
| **Violation** | `allocated + unlocked + reserved + consumed + expired ≠ initial_allocation` |
| **Detection** | SQL CHECK constraint on `finn_credit_accounts` table (`credit-persistence.ts:44-46`). In-memory: `defaultConservationCheck()` at `consumption.ts:74-78`. SQL: `verifyConservationSQL()` at `credit-persistence.ts:215`. |
| **Recovery** | Postgres rejects the UPDATE — transaction rolls back. In-memory state unchanged. ERROR logged. Credit operation fails atomically. |
| **Blast radius** | Single credit account. Transaction rollback prevents partial state. Other accounts unaffected. |

### Nonce Replay Protection

| Field | Description |
|-------|-------------|
| **Violation** | Same nonce used twice for unlock/consume within TTL |
| **Detection** | `finn_used_nonces` table with TTL-based uniqueness (`nonce-cleanup.ts`). Duplicate insert fails. |
| **Recovery** | Operation rejected. Nonces pruned after TTL expiry to bound storage. |
| **Blast radius** | Single credit operation rejected. No state corruption. |
