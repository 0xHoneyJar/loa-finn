## WAL Invariants

<!-- provenance: CODE-FACTUAL -->
<!-- evidence: file=src/persistence/wal.ts, file=src/billing/state-machine.ts:54-77 -->

### Monotonic Sequence Numbers
> WAL sequence numbers are strictly monotonically increasing within a single process
- **Source**: `src/billing/state-machine.ts:65-66` — `nextWALSequence()` increments global counter
- **Multi-process**: Each process has independent sequence space; replay orders by timestamp + process ID

### CRC32 Checksum Validation
> Every WAL envelope includes CRC32 of payload for corruption detection
- **Source**: `src/billing/state-machine.ts:46-51`
- Computed at append, verified on replay

### Atomic Segment Rotation
> Current segment file is never partially visible; rename is atomic
- **Source**: `src/persistence/wal.ts:91-92`
- Pattern: write to temp, rename atomically — prevents torn writes on crash

### Disk Pressure Hysteresis
> Enters read-only at 100MB free, resumes at 150MB free
- **Source**: `src/persistence/wal.ts:43-45, 75-88`
- LOW: 100MB threshold, HIGH: 150MB threshold — prevents thrashing at boundary

### WAL Authoritativeness
> WAL is the source of truth. Redis is derived, never authoritative.
- **Source**: `src/billing/state-machine.ts:1-6`
- All billing state transitions write WAL first, then update Redis cache
