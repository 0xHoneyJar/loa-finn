# loa-finn — Architecture Reference

> Generated by Ground Truth v1.0.0

## System Layers

<!-- provenance: CODE-FACTUAL -->
<!-- evidence: file=src/index.ts:2-863 -->
loa-finn is a 15-module modular monolith organized in 5 layers. Boot sequence follows strict dependency order: config → identity → persistence → recovery → gateway → scheduler → serve.

### Layer 1: Configuration & Formats
- **Entry**: `loadConfig()` at `src/config.ts`
- **Boot**: `src/index.ts:43`
- TypeBox format registration (`src/hounfour/typebox-formats.js`) runs as global singleton before any `Value.Check` calls
- `FinnConfig` interface spans 150+ environment variables across all subsystems

### Layer 2: Identity & Upstream Framework
- **Entry**: `IdentityLoader` at `src/index.ts:58-64`
- Loads BEAUVOIR personality from `BEAUVOIR_PATH`
- FileWatcher hot-reload on document changes (`src/index.ts:707-711`)
- Upstream persistence from `.claude/lib/persistence/` re-exported via `src/persistence/upstream.ts`

### Layer 3: Persistence (WAL-based)
- **Entry**: `createWALManager(walDir)` at `src/index.ts:88`
- ULID-sequenced append-only log with CRC32 checksums
- Segment rotation at 10MB threshold
- Recovery cascade: template → R2 → git (`src/persistence/recovery.ts`)
- Disk pressure hysteresis: read-only at 100MB free, resume at 150MB

### Layer 4: Protocol & Billing (Hounfour)
- **Multi-model router**: `src/hounfour/router.ts` — ProviderRegistry, BudgetEnforcer, health prober, rate limiter
- **Goodhart protection**: `src/hounfour/goodhart/` — states: disabled, shadow, enabled, init_failed
- **Billing state machine**: `src/billing/state-machine.ts` — IDLE → RESERVE_HELD → COMMITTED → FINALIZE_PENDING → FINALIZE_ACKED
- **S2S JWT**: ES256 in production (HS256 blocked unless explicit), JTI replay protection
- **Audit trail**: Buffered chain with KMS signing, dual-format (legacy + chain-bound v8.3.1)

### Layer 5: Gateway & Scheduling
- **HTTP**: Hono v4 at `src/gateway/server.ts` — 25+ endpoints across 8 route groups
- **WebSocket**: `src/gateway/ws.ts` — real-time agent sessions with event streaming
- **Scheduler**: 4 cron tasks (R2 sync, git sync, health, WAL prune) at `src/index.ts:642-701`

## Data Flow: Invoke Path

```
HTTP POST /api/v1/invoke
  → hounfourAuth middleware (JWT → TenantContext)
  → rate limiter check (per-tenant RPM/TPM)
  → economic boundary evaluation (tier → trust → capital)
  → billing reserve (WAL append → Redis cache → governed shadow)
  → Goodhart routing decision (provider selection)
  → provider invoke (Anthropic/OpenAI/Cheval)
  → billing commit (actual_cost → WAL → DLQ enqueue)
  → response serialization (cost_micro, trace_id, knowledge)
```

## Persistence Architecture

<!-- provenance: CODE-FACTUAL -->
<!-- evidence: file=src/persistence/wal.ts, file=src/persistence/r2-sync.ts, file=src/persistence/git-sync.ts, file=src/persistence/pruner.ts -->

| Component | Role | Durability |
|-----------|------|------------|
| WAL (local) | Authoritative append-only log | Disk (fsync) |
| R2 sync | Off-site backup | S3-compatible object store |
| Git sync | Snapshot backup | Git repository |
| Redis | Derived cache (billing, sessions) | AOF if configured |
| PostgreSQL | Credits, conversations, NFT metadata | ACID transactions |
| Pruner | Compaction of confirmed segments | Runs hourly |

**Key invariant**: WAL is authoritative. Redis is derived, never the source of truth (`src/billing/state-machine.ts:1-6`).

## Security Architecture

<!-- provenance: CODE-FACTUAL -->
<!-- evidence: file=src/hounfour/jwt-auth.ts, file=src/gateway/routes/keys.ts -->

| Auth Method | Where Used | Mechanism |
|-------------|------------|-----------|
| SIWE JWT | Conversations, API keys | Sign-In with Ethereum → JWT |
| S2S JWT (ES256) | Hounfour tenant API | JWKS verification, JTI replay guard |
| API key (hashed) | Oracle product | Redis-backed, bcrypt-hashed |
| Bearer token | Sessions, metrics | Shared secret (`FINN_AUTH_TOKEN`) |
| HMAC-SHA256 | Cheval sidecar | Per-request signature with nonce |
| x402 payment | Pay-per-request | On-chain payment verification |

**Production constraints**:
- ES256 required for S2S JWT (`src/index.ts:486-491`) — HS256 blocked unless explicit
- JTI replay window: 600s default (`src/hounfour/jwt-auth.ts:29-32`)
- Issuer allowlist enforced (`src/hounfour/jwt-auth.ts:115-116`)

## Key Design Decisions

<!-- provenance: CODE-FACTUAL -->

| Decision | Rationale | Evidence |
|----------|-----------|----------|
| WAL over direct DB | Crash-deterministic recovery, decoupled from network partitions | `src/persistence/wal.ts` |
| Dual auth (SIWE + S2S) | SIWE for end-users (wallet-native), S2S for service mesh | `src/hounfour/jwt-auth.ts` |
| Billing DLQ pattern | At-least-once finalization despite network failures | `src/hounfour/dlq-store.ts` |
| Non-fatal boot | Pod always reaches READY; subsystems degrade independently | `src/index.ts` (26-step boot) |
| Goodhart shared holder | Atomic routing state propagation without restart | `src/index.ts:230-236` |
| Chain-bound audit hashes | Tamper-evident ordering — hash includes previous hash | `src/hounfour/protocol-types.ts` |
